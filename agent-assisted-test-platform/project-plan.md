# Agent 辅助测试平台实习项目计划

## 1. 项目背景

团队现有内网测试平台源码位于 `radiatest`。由于核心成员离职和长期缺少持续维护，Gitee 登录、任务管理、用例管理等功能已经出现不同程度腐化。

代码走查显示，旧平台包含四类服务、二十多个业务域，前后端合计约 11.5 万行，并依赖数据库、Redis、RabbitMQ、Celery、PXE、IPMI 和 libvirt。依赖与部署链路老化，现有自动化测试也不足以支撑安全改造。

因此，`radiatest` 只作为领域知识、数据字段和基础设施流程参考；本项目不承担“修好全部旧平台”或“等价重写全部功能”。

## 2. 方案评估与结论

| 方案 | 优点 | 主要问题 | 结论 |
|---|---|---|---|
| 修复旧项目 | 保留历史能力，局部问题可能较快修复 | 耦合重、回归保护弱，实习成果容易退化为零散补丁 | 不选 |
| 完整重写旧平台 | 可以重新设计架构 | 三个月无法完成等价替代，成功标准不可控 | 不选 |
| 新建项目并分期交付 | 一期可形成独立闭环，并建立新的测试、部署和文档基线 | 需要明确迁移与阶段边界 | 采用 |

> 采用方案 C。新建独立项目，三个月必须交付机器资源基础管理；二期只在一期稳定后尝试最小技术闭环；IM 集成、用例管理和任务管理分别作为三、四、五期路线。

## 3. 总体目标

建设一个单团队使用、可持续演进的内网测试平台，最终覆盖：

```text
机器资源台账与占用
  -> 虚机/物理机生命周期操作
  -> 飞书等 IM 与 LLM 助手集成
  -> 测试用例管理
  -> 测试任务执行与结果管理
```

本次实习周期按三个月设计：

- 必须完成：一期机器资源基础管理，并完成真实用户试用。
- 争取完成：二期中的一个最小技术闭环，例如受控远程命令或单台 KVM 虚机创建。
- 路线规划：完整描述二至五期，但不把三至五期计入本次实习验收。

## 4. 项目边界

### 4.1 一期：机器资源基础管理

一期覆盖以下能力：

1. 用户与权限

   - 本地账号登录。
   - 账号由 `ADMIN` 创建、禁用和重置密码。
   - 不开放自助注册。
   - 不接入 Gitee OAuth。
   - 固定三类角色：`ADMIN`、`TSE`、`TE`。
   - 所有平台用户可自行创建带名称和有效期的个人访问令牌（PAT），用于脚本或独立客户端调用 API。
   - PAT 明文只在创建时展示一次，数据库仅保存不可逆哈希；用户和 `ADMIN` 均可撤销，`ADMIN` 不能查看令牌明文。

2. 资源池与机器台账

   - 创建和维护资源池。
   - 资源池表达实验室、网络区域或用途，不承载多组织权限。
   - 统一纳管已有物理机和虚机，以资源类型区分 `PHYSICAL` 和 `VIRTUAL`。
   - 虚机可关联宿主物理机。
   - 每台机器具有不可变的 `resource_code`：物理机使用设备 SN，虚机使用创建时生成并与虚拟化层保持一致的标准 UUID。
   - 资源名称用于搜索和展示；虚机名称可按版本、架构和时间生成，但名称、IP 和 MAC 均不作为写操作的稳定标识。
   - 支持标签、负责人、架构、操作系统、IP、MAC、BMC、SSH 等必要信息。
   - 支持按资源池、类型、标签、状态、负责人等条件查询。

3. 占用与租约

   - 用户可立即占用空闲机器，无需逐单审批。
   - 同一机器不能被重复占用。
   - 资源租约必须记录用途、开始时间和结束时间。
   - 占用 API 接收 `duration_minutes`，由服务端计算起止时间并校验最大租期；核心 API 不解析“2 小时”等自然语言，也不依赖客户端时区计算结束时间。
   - 支持释放本人占用的机器。
   - 支持到期自动释放或进入明确的到期处理状态。
   - `TSE` 和 `ADMIN` 可处理延期、冲突和强制释放。

4. 凭据管理

   - 保存 SSH、BMC 等连接凭据。
   - 凭据使用应用密钥加密后入库，不保存明文。
   - 用户点击显示时，后端鉴权后实时解密并短暂返回。
   - 前端默认掩码，不在浏览器持久化明文。
   - 普通机器凭据仅向有权限且满足使用条件的用户展示。
   - 标记为关键机器的资源，凭据只有 `ADMIN` 可查看。
   - 每次查看凭据必须写入审计日志。

5. 连通性检查

   - 提供按需执行的 Ping 或 SSH 端口连通性检查。
   - 展示最近一次检测时间和结果。
   - 一期不建设常驻监控系统。

6. 数据导入与迁移

   - 支持 CSV 模板导入和校验报告。
   - 提供可重复执行的示例数据初始化方式。
   - 新模型稳定后，提供从 `radiatest` 只读导出并一次性导入新平台的迁移脚本。
   - 新旧系统不共用数据库，不要求双向同步。

7. 审计与可运维性

   - 记录登录、机器变更、占用、释放、强制操作和凭据查看等关键事件。
   - 提供基础审计查询。
   - 支持单机 Linux + Docker Compose 部署。
   - 提供健康检查、结构化日志、数据库迁移和备份恢复说明。

8. API 与外部集成

   - 一期全部业务能力必须提供稳定、可文档化的 API，包括用户与 PAT、资源池、机器台账、CSV 导入、占用、释放、延期、连通性检查、授权后的凭据查看和审计查询。
   - Web 和未来外部集成都调用同一组业务 API；一期不实现任何具体 IM 适配器。
   - 占用 API 使用 `resource_code` 指定机器并返回不可变 `lease_id`；释放和延期 API 必须使用 `lease_id` 并校验租约归属，避免延迟请求作用于下一位占用者。
   - 占用成功响应只返回资源租约、机器信息和租期，不直接返回 SSH/BMC 密码；凭据通过独立 API 读取，以便单独鉴权和审计，并避免幂等响应存储凭据明文。
   - 查询 API 支持按名称、IP、MAC、SN 等条件搜索；所有 API 均执行与 Web 一致的权限、并发和审计规则。
   - API 使用 `/api/v1` 主版本、稳定错误码、`request_id`、OpenAPI 文档和向后兼容检查。
   - 用户 PAT 继承所属用户的当前角色权限；所有写 API 支持 `Idempotency-Key`。
   - 一期占用 API 每次只处理一台机器并创建一条资源租约；不提供自动选择 N 台机器或批量占用事务。
   - 一期持久化资源租约关键事件，至少包含占用、延期、释放、到期和强制释放，并提供带游标的事件查询 API。

### 4.2 一期明确不做

- 不创建、删除或修改虚机。
- 不执行 PXE 装机。
- 不执行远程 Shell 命令。
- 不控制物理机或虚机开关机。
- 不管理测试用例。
- 不管理或调度测试任务。
- 不实现多组织、多租户和复杂组织树权限。
- 不引入微服务、消息队列或 Kubernetes 作为一期前置条件。
- 不接入 Gitee OAuth、企业 SSO 或其他外部身份系统。
- 不实现飞书机器人或其他具体 IM 应用。
- 不与 `radiatest` 共用运行时服务或数据库。

## 5. 权限边界

### 5.1 角色职责

| 能力 | ADMIN | TSE | TE |
|---|---|---|---|
| 管理用户和角色 | 是 | 否 | 否 |
| 管理系统配置 | 是 | 否 | 否 |
| 创建和维护资源池 | 是 | 是 | 否 |
| 登记和维护机器 | 是 | 是 | 否 |
| 查看机器公共信息 | 是 | 是 | 是 |
| 占用空闲机器 | 是 | 是 | 是 |
| 释放本人占用 | 是 | 是 | 是 |
| 强制释放他人占用 | 是 | 是 | 否 |
| 查看普通机器凭据 | 是 | 按授权 | 仅本人占用且按授权 |
| 查看关键机器凭据 | 是 | 否 | 否 |
| 查看完整审计记录 | 是 | 是 | 否 |

技术设计阶段需要把此表细化为接口级权限矩阵，并通过自动化测试验证。默认采用最小权限；任何额外授权都必须在 ADR 或需求记录中说明。

### 5.2 状态设计要求

实习生需在领域设计中避免使用一个字段混合表达多种状态，至少区分：

- 管理状态：启用、维护、停用等。
- 连通状态：未知、可达、不可达等。
- 占用状态：空闲、已占用、已到期等。

具体命名和数据库结构由实习生设计并在评审中确认。

## 6. 技术选型与架构评审

本计划不写死具体框架。技术选型是实习生的第一项正式设计任务。

### 6.1 必须满足的架构约束

- 优先采用模块化单体，不在一期拆分微服务。
- 使用关系型数据库和正式数据库迁移机制。
- 前后端接口有稳定契约和自动生成或可维护的 API 文档。
- 采用 API-first 方式：核心业务规则位于应用服务或领域层，Web 和外部集成只是调用入口。
- 本地开发和验收环境可容器化启动。
- 核心领域逻辑可脱离外部基础设施进行测试。
- 后续资源执行代理可以独立部署到不同内网资源池。
- 用户身份、个人 PAT 和集成客户端凭据必须分离建模，禁止通过共享管理员令牌实现外部渠道接入。
- 凭据加密密钥与数据库分离配置。
- 不直接依赖 `radiatest` 的 Python 包、数据库表或在线服务。

### 6.2 技术选型任务

实习生需比较 2—3 套可行方案，至少说明：

- 前端、后端和数据库候选方案。
- 团队熟悉度与学习成本。
- 权限、迁移、测试和 API 支持。
- 异步作业和资源执行代理的后续演进方式。
- 部署、升级、故障排查和长期维护成本。
- 主要风险和放弃其他方案的原因。

交付一份简短 ADR，经导师评审通过后，才能进入主体开发。评审关注决策质量，不要求选择预设答案。

## 7. Agent 辅助研发方式

### 7.1 Agent 可参与的工作

- 检索和解释 `radiatest` 旧代码。
- 从旧系统提炼字段、业务规则和失败模式。
- 辅助技术方案比较和 ADR 初稿。
- 拆分 issue、生成代码草案和测试草案。
- 辅助代码审查、错误定位和回归分析。
- 同步 API、部署、迁移和交接文档。

### 7.2 使用要求

- 不以 Agent 调用次数、生成代码行数或 Prompt 数量作为成绩。
- 实习生必须能解释自己提交的设计、代码和测试。
- Agent 生成的改动必须经过人工审查并运行匹配范围的验证。
- 不向外部模型提交真实密码、私钥、令牌和未脱敏内网信息。
- 重要架构和安全决策必须形成 ADR，不能只保留在对话记录中。
- 每周周报记录 Agent 参与点、人工验证方式、错误建议和修正过程。

### 7.3 推荐工程上下文

新代码仓建议至少维护：

- `README.md`：本地启动、测试和部署入口。
- `AGENTS.md`：Agent 在该仓库中的操作约束。
- `CONTEXT.md`：领域语言、核心流程和系统边界。
- `docs/adr/`：技术和架构决策。
- `docs/api/` 或自动生成 API 文档。
- `docs/runbook/`：部署、升级、备份恢复和故障处理。

## 8. 12 周实施计划

### 第 1 周：旧系统走查与需求收敛

目标：

- 理解现有机器管理流程和真实使用痛点。
- 明确一期边界、角色和验收方式。

任务：

- 在导师协助下走查 `radiatest` 的机器、用户和权限模块。
- 访谈至少 2 名实际用户，记录机器登记、查询、占用、释放和凭据使用流程。
- 绘制现状流程和一期目标流程。
- 选取 5 条关键用户旅程作为端到端验收骨架。
- 建立项目风险清单和术语表。

交付：

- `legacy-assessment.md`。
- `requirements.md`。
- `domain-glossary.md`。
- 5 条关键用户旅程。

验收：

- 能说明为什么不直接修复或等价重写旧平台。
- 能清楚讲出一期做什么、不做什么。
- 用户旅程得到导师确认。

### 第 2 周：技术选型、领域模型与工程骨架

目标：

- 完成技术选型评审。
- 建立可持续开发的最小工程基线。

任务：

- 对比 2—3 套技术方案并提交 ADR。
- 设计用户、角色、资源池、机器、凭据、资源租约和审计的领域关系。
- 明确 API 约定、错误格式、配置方式和迁移策略。
- 定义首批稳定业务错误码，例如资源不存在、租约冲突、无权释放和凭据不可见。
- 建立 OpenAPI 基线和破坏性变更检查。
- 明确写 API 的幂等键作用域、保存期限、重复响应和参数冲突行为。
- 建立代码仓、自动化检查、测试入口和 Docker Compose 开发环境。
- 用一个最小接口和页面证明端到端链路可运行。

交付：

- 技术选型 ADR。
- 领域模型图和状态说明。
- API 设计草案。
- OpenAPI 初稿和最小调用示例。
- 可启动的工程骨架。

验收闸门：

- 导师评审通过技术选型和领域模型后，才能进入完整功能开发。
- 新成员能根据 README 启动项目并运行第一条测试。

### 第 3 周：本地账号与 RBAC

目标：

- 完成本地登录和三角色权限基线。

任务：

- 实现 `ADMIN` 创建、禁用、重置账号。
- 实现安全的密码散列和登录会话。
- 实现 PAT 创建、列表、过期和撤销，并确保令牌明文只展示一次。
- 建立 `ADMIN`、`TSE`、`TE` 接口级权限矩阵。
- 完成前后端路由保护。
- 为权限矩阵编写自动化测试。

交付：

- 用户管理页面和 API。
- 权限矩阵文档。
- 登录与权限自动化测试。
- PAT 生命周期和角色继承测试。

验收：

- 三类角色只能看到并执行授权范围内的操作。
- 禁用用户不能继续登录。
- 用户禁用、角色变更或 PAT 撤销后，对应 API 调用立即失效。
- 权限测试能够阻止越权回归。

### 第 4 周：资源池与机器台账

目标：

- 跑通机器资源登记、维护、查询和展示。

任务：

- 实现资源池管理。
- 实现物理机和虚机统一台账。
- 实现虚机与宿主机关联。
- 实现标签、关键机器标记和常用筛选。
- 区分管理、连通和占用状态。

交付：

- 资源池与机器管理页面。
- 机器管理 API。
- 数据库迁移和领域测试。

验收：

- 能登记并查询真实的物理机和已有虚机。
- 能准确表达虚机宿主关系。
- 物理机 SN 和虚机 UUID 在各自资源类型内唯一且创建后不可修改。
- 关键查询和唯一性约束有自动化测试。

### 第 5 周：CSV 导入与旧数据迁移验证

目标：

- 降低现有资源进入新平台的成本。

任务：

- 定义 CSV 模板和字段校验规则。
- 实现预检查、错误报告和确认导入。
- 设计可重复执行且不会产生重复数据的导入策略。
- 从 `radiatest` 抽取一小批脱敏数据验证映射。
- 记录不迁移字段及原因。

交付：

- CSV 模板与说明。
- 导入工具和错误报告。
- `radiatest` 字段映射文档。
- 小批量迁移验证记录。

验收：

- 错误数据不会部分写入并造成脏状态。
- 重复导入行为明确且可验证。
- 至少一批真实机器数据成功进入新平台。

### 第 6 周：占用、租约与并发冲突

目标：

- 完成机器资源使用闭环。

任务：

- 实现即时占用、本人释放、延期和强制释放。
- 实现最长租期配置。
- 统一由服务端根据 `duration_minutes` 计算租约结束时间。
- 处理两个用户同时占用同一机器的并发冲突。
- 验证占用、释放和延期在重复请求下不会重复变更资源租约。
- 验证旧租约的延迟释放或延期请求不能影响同一机器上的新租约。
- 实现到期处理机制和必要通知提示。
- 持久化占用、延期、释放、到期和强制释放事件，并提供可增量查询的事件 API。
- 记录占用和释放审计事件。

交付：

- 占用与租约页面和 API。
- 状态转换说明。
- 租约事件说明和事件查询调用示例。
- 并发和到期处理测试。

验收：

- 同一时刻一台机器只能有一个有效占用者。
- 普通占用无需人工审批。
- 非法、超过上限或小于最小值的租期被稳定错误码拒绝。
- 越权释放、重复释放和过期操作有明确结果。
- 每次有效租约变化产生且只产生一条可追踪事件，调用方可按游标连续读取。

### 第 7 周：凭据加密与审计

目标：

- 在保持使用便利的同时建立最低安全基线。

任务：

- 实现 SSH、BMC 凭据加密入库。
- 实现后端鉴权后的实时解密查看。
- 实现关键机器仅 `ADMIN` 可查看凭据。
- 避免凭据出现在普通日志、异常堆栈和前端持久化存储中。
- 完成凭据查看审计。

交付：

- 凭据管理设计说明。
- 加密、轮换和恢复说明。
- 权限和审计测试。

验收：

- 直接读取数据库无法获得明文凭据。
- 未授权角色无法通过接口或页面看到凭据。
- 每次查看都有可查询的审计记录。

### 第 8 周：按需连通性检查与一期体验完善

目标：

- 补齐资源使用前的基本可用性判断。
- 完成一期端到端主流程。

任务：

- 实现 Ping 或 SSH 端口按需检查。
- 保存最近检测时间、结果和错误摘要。
- 优化资源列表、详情、筛选和占用入口。
- 跑通 5 条关键用户旅程。
- 处理前 8 周积累的高优先级问题。

交付：

- 连通性检查能力。
- 一期端到端演示。
- 第一版用户指南。

验收：

- 用户能从查询机器一直走到占用、查看凭据和释放。
- 检查失败不会误改机器占用状态。
- 关键流程具备端到端测试。

### 第 9 周：部署、质量加固与一期发布候选版

目标：

- 形成可部署、可恢复、可交接的一期候选版本。

任务：

- 完善 Docker Compose 部署。
- 补齐健康检查、日志和配置说明。
- 验证数据库迁移、备份和恢复。
- 补齐核心领域测试和权限测试。
- 修复安全、并发和数据一致性问题。

交付：

- 一期发布候选版。
- 部署、升级、备份恢复文档。
- 自动化测试报告。
- 已知限制清单。

验收闸门：

- 一期验收项全部通过后，才能把开发时间投入二期技术验证。

### 第 10 周：真实用户试用启动

目标：

- 用真实机器和用户验证一期主流程。

任务：

- 导入一批真实机器并部署候选版本。
- 邀请 3—5 名用户开始连续一周试用。
- 记录成功路径、失败路径、人工绕行和高优先级问题。
- 阻断性问题立即修复并回归。

交付：

- 试用环境和用户清单。
- 每日问题与修复记录。

验收：

- 真实用户能完成查询、占用、查看凭据和释放主流程。
- 所有阻断性问题都有负责人和处理结论。

### 第 11 周：试用修复与一期验收

目标：

- 完成一周试用并形成一期正式候选版本。

任务：

- 按严重程度修复试用问题并完成回归。
- 复核一期功能、权限、API、迁移和部署验收项。
- 邀请非作者按文档完成一次干净部署。
- 形成已知限制和后续问题清单。

交付：

- `pilot-report.md`。
- 一期正式候选版本。
- 自动化测试和验收报告。

验收：

- 至少有一周真实试用记录，核心流程无阻断性问题。
- 一期所有必须完成项通过验收。

### 第 12 周：交接与可选二期验证

目标：

- 完成交接；仅在一期全部验收通过后尝试二期最小技术验证。

任务：

- 整理代码、测试、迁移、部署和运维文档。
- 复核文档与实际行为一致。
- 输出一期复盘、二期 backlog 及三至五期路线。
- 若时间允许，从 libvirt、PXE、IPMI 或远程命令中选择一个风险可控场景做技术验证；不得影响一期交接。
- 准备 15 分钟以内的真实系统演示。

交付：

- 一期正式交付版本。
- `handover.md`。
- `known-limitations.md`。
- `phase-2-backlog.md`。
- 项目复盘和演示材料。
- 可选：二期架构 ADR 和一个最小技术验证。

最终验收：

- 新成员可以根据文档启动、测试和部署平台。
- 一期核心流程、权限和迁移均可复现。
- 团队明确后续入口，不依赖实习生口头说明。

## 9. 一期验收标准

### 9.1 必须完成

- 独立新项目和数据库，不依赖 `radiatest` 在线运行。
- 本地账号登录及 `ADMIN`、`TSE`、`TE` 权限控制。
- 可撤销、可过期且仅保存哈希的用户 PAT。
- 单团队、单组织边界。
- 资源池管理。
- 物理机和已有虚机统一台账。
- 标签、关键机器、宿主关系和筛选查询。
- 即时占用、租约、延期、释放、强制释放和并发冲突处理。
- SSH/BMC 凭据加密保存、实时授权查看和审计。
- CSV 导入及一批真实机器迁移验证。
- 按需连通性检查。
- Docker Compose 部署、数据库迁移、备份恢复和运行手册。
- 3—5 名用户连续一周试用及试用报告。

### 9.2 自动化质量门槛

- 核心后端领域逻辑测试覆盖率不低于 80%。
- 权限矩阵、PAT 生命周期和用户状态变化均有自动化测试。
- 登录、机器导入、占用冲突、到期处理和凭据查看至少各有一条端到端测试。
- 占用、释放和延期的契约测试覆盖权限、幂等、旧租约竞态和审计行为。
- 租约事件 API 覆盖连续读取、权限和重复读取；典型失败场景返回稳定错误码及 `request_id`。
- OpenAPI 兼容检查能阻止 `/api/v1` 未声明的破坏性变更，并有脱离 Web 的调用示例。
- 按功能清单验证 Web 与 API 能力一致，不存在绕过业务 API 的页面专属逻辑。
- 自动化检查在干净环境中可以重复运行。
- 部署文档由非作者实际走通一次。

### 9.3 最好完成

- 完成从 `radiatest` 到新平台的一次性迁移脚本。
- 完成二期中的一个最小技术验证。
- 增加基础使用指标和异常告警。

### 9.4 不强求完成

- 完整二期。
- 测试用例管理。
- 测试任务管理。
- Kubernetes 部署。
- 企业 SSO 或 OAuth。
- 多租户和复杂审批流。

## 10. 后续阶段路线图

### 10.1 二期：机器生命周期与远程操作

目标：在一期资源台账、权限和资源租约模型之上，安全地执行基础设施操作。

首批只支持现有环境：

- KVM/libvirt：创建、删除、启动、停止和查询虚机。
- PXE：物理机自动装机。
- IPMI：物理机开机、关机和重启。
- SSH：受控的一次性远程命令执行。

设计要求：

- 按资源池部署独立执行代理，避免平台主服务直接进入所有内网。
- 所有操作统一建模为异步作业，包含作业编号、状态、进度、超时、错误和审计。
- 操作应具备幂等性或明确的重复提交保护。
- 创建虚机时生成标准 UUID 作为 `resource_code` 并同步给虚拟化层；版本、架构和时间只用于生成可读资源名称。
- 删除虚机、装机、强制关机等破坏性操作必须二次确认。
- `ADMIN/TSE` 可操作有权限的机器；`TE` 只能操作自己占用的机器。
- 远程命令记录操作者、目标、命令、开始时间、结束时间、退出码和输出摘要。
- 首版不做交互式 Web 终端。
- 首版不同时兼容 OpenStack、VMware、Proxmox 等其他平台。

二期完成标志：

- 至少一个资源池稳定接入执行代理。
- libvirt、PXE、IPMI 和 SSH 操作均有成功、失败、超时和审计闭环。
- 操作失败不会让资源台账长期停留在无法解释的状态。

### 10.2 三期：飞书等 IM 与 LLM 助手集成

目标：基于一期业务 API 和租约事件，将机器资源能力接入飞书，并通过可替换的 OpenAI-compatible LLM API 提供自然语言交互；渠道和模型均不承载平台业务规则。

范围：

- 飞书适配器使用独立集成客户端凭据；用户通过一次性绑定码关联平台账号，所有操作按真实用户授权和审计。
- 支持查询、我的租约、占用、释放、延期、连通性检测、凭据查看和租约通知；IM 重试复用幂等键。
- 普通机器凭据可在单聊中返回给当前占用者，群聊拒绝，关键机器仍仅 `ADMIN` 可查看。
- Web 与飞书共用认证后的助手 API、命令注册表、Prompt、工具定义和操作草案模型。
- `help`/`帮助` 按角色展示完整快捷命令卡片；明确命令和参数补全走确定性流程，只有未命中命令的自由文本进入 LLM。
- 首版启用一个由 `ADMIN` 配置的 OpenAI-compatible 服务，必须支持 tool calling；三个角色均可使用，但工具和结果始终受当前用户权限约束。
- LLM 首版只做自然语言资源检索、占用/释放/延期草案，以及错误、当前租约和平台用法解释。
- 只读工具可直接执行；写工具只生成短时操作草案，须由同一用户在原渠道和会话中明确确认后执行。
- 平台负责单位换算、Schema、唯一性、权限、状态和幂等校验；歧义或非法工具参数必须澄清，不得猜测。
- 平台帮助只检索仓库内文档；工具不返回密码、令牌或密钥；LLM 故障时回退普通表单和搜索。
- 不长期保存普通问答，只保留工具调用、草案、确认和执行结果。

详细取舍见 `docs/adr/`，领域术语见 `CONTEXT.md`。

首版不做：

- 不接入第二个 IM，不做多模型路由、自动切换或用户选模型。
- 不让适配器或 LLM 直接访问数据库、复制业务规则或绕过业务 API。
- 不让 LLM 读取凭据、生成远程 Shell、自主调度资源或自动执行写操作。
- 不建设向量数据库、知识库后台或跨渠道草案确认。

三期完成标志：

- 已绑定用户可以在飞书单聊中查询、占用、延期和释放机器。
- 未绑定用户、越权用户和群聊凭据请求均被明确拒绝。
- Web 与飞书具有一致的权限、幂等、快捷命令、草案确认和审计语义。
- 操作草案可检查、可追溯；过期、跨用户、跨渠道或跨会话确认均被拒绝。
- 使用目标 OpenAI-compatible 服务和一个可控 Mock 验证 tool calling、参数校验、多轮工具结果回传和异常降级。
- LLM 故障不影响普通查询与资源操作，配置密钥不可重新读取明文。
- 租约到期等事件能够可靠通知对应用户，并能追踪失败投递。
- 至少 30 条固定样本覆盖检索、草案、歧义、错误和越权；未确认写操作、越权调用及密码/令牌泄漏成功次数均为 0。

### 10.3 四期：测试用例管理

目标：管理结构化测试用例，同时保留自动化脚本在 Git 中的工程属性。

范围：

- 结构化用例内容存入平台数据库。
- 支持手工用例和自动化用例两种类型。
- 自动化用例记录 Git 仓库、分支、脚本路径和运行参数，不在平台内重建 Git 托管。
- 支持目录或用例集、标签、搜索、复制和导入导出。
- 支持 `DRAFT -> ACTIVE -> DEPRECATED` 生命周期。
- 保存必要的修改历史。
- 由 `TSE` 发布或废弃用例。

首版不做：

- Git 代码编辑器。
- Git 分支、提交和合并请求替代品。
- 多人会签、复杂评审流和评论线程。

四期完成标志：

- 用户可以创建、维护、检索和发布结构化用例。
- 自动化用例可以稳定定位到对应 Git 脚本。
- 用例变更可追踪，废弃用例不会继续被新任务选择。

### 10.4 五期：测试任务管理

目标：将机器、用例和二期执行器串成测试执行与结果闭环。

范围：

- 一个任务选择若干机器和若干用例。
- 用户手动启动任务。
- 自动化用例通过二期执行器下发并回传日志与结果。
- 手工用例由执行人填写 `PASS`、`FAIL`、`BLOCKED`、备注和附件。
- 支持运行状态、进度、取消、失败重试和结果汇总。
- 保留任务、运行批次和单次用例结果之间的清晰关系。

首版不做：

- DAG 编排。
- 定时任务。
- 复杂资源调度策略。
- 多阶段 CI/CD 流水线。

五期完成标志：

- 能从选择机器和用例开始，完成一次手工或自动化测试批次。
- 每个结果都能追溯到任务、用例、机器、执行人和日志。
- 失败任务可以取消或重试，且不会重复占用资源或丢失历史记录。

## 11. 主要风险与控制

| 风险 | 控制方式 |
|---|---|
| 重新走向完整重写 | 每期明确非目标，新需求必须归入当前期或后续路线 |
| 技术选型反复 | 比较限时一周，只评估 2—3 套方案，评审后形成 ADR |
| 过度依赖 Agent | 要求解释提交内容，以测试和运行结果验收，周报记录错误建议及修正 |
| 凭据泄漏 | 加密入库、密钥分离、查看鉴权审计，测试和日志不得包含真实凭据 |
| 并发占用冲突 | 数据库事务与约束保证单一有效租约，并覆盖并发测试 |
| 旧数据迁移拖期 | 先定新模型，再做 CSV 小批量验证；不追求无损迁移所有字段 |
| 二期操作破坏环境 | 一期验收后使用专用资源，仅验证一种能力，并启用 dry-run、超时和审计 |

## 12. 最终结论

推荐路线：

```text
新建独立项目
  -> 技术选型与架构评审
  -> 机器资源台账、权限、资源租约、凭据与审计
  -> 真实用户试用
  -> libvirt/PXE/IPMI/SSH 生命周期能力
  -> 飞书等 IM 与 LLM 助手集成
  -> 结构化测试用例
  -> 测试任务与结果闭环
```

一句话总结：

> 不让实习生接盘整个旧平台，而是借助 Agent 从真实机器管理痛点切入，三个月交付可试用的一期，并为后续资源执行、用例管理和任务管理建立清晰、可验证的演进路径。
